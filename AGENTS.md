## Prohibited at a glance

Quick-scan list of hard bans across the codebase — see the categorized sections below for
full context on each.

- No email, phone number, real name, or any PII field — anywhere, ever (signup, recovery,
  analytics, Keycloak claims).
- No third-party identity provider, auth-as-a-service, or managed cloud database.
- No Docker dependency (Podman only); no manual local-dev setup steps outside the Aspire
  AppHost.
- No UI library other than Angular Material; no Tailwind/Bootstrap/utility-class CSS.
- `::ng-deep`, `[style]` inline bindings, and hard-coded hex/font/shadow values outside M3
  tokens — all PROHIBITED in Angular code.
- No raw SQL (`FromSqlRaw`, `ExecuteSqlRaw`, etc.) — EF Core LINQ only.
- No second/NoSQL database engine — PostgreSQL `jsonb` covers document-shaped data.
- No AutoMapper/Mapster/TinyMapper or other convention-based object mapping.
- No direct frontend → backend API calls — Angular talks to the BFF only.
- No HTTP polling as a substitute for SignalR push delivery.
- No hard deletes on any entity — soft-delete (`IsActive`) only.
- No test files, test projects, or test libraries of any kind (`.spec.ts`, `*Tests.csproj`,
  `tests/`, `__tests__/`, xUnit/NUnit/MSTest/Jest/Jasmine/Karma/Playwright, etc.).
- No unjustified warning/lint suppressions (`#pragma warning disable`, `[SuppressMessage]`,
  `// @ts-ignore`, `// eslint-disable`) or bare `!` null-forgiving operator — every suppression
  needs an inline comment proving it's safe.
- No commented-out code or `// TODO` comments merged to main.
- No quality shortcuts adopted from memory/subjective preference when official documentation
  exists and wasn't checked.

## Authentication & privacy

- Signup requires **exactly one credential: a password**. No email, phone number, real name, or
  any other PII, ever — not at signup, not for recovery, not for analytics.
- Username is **autogenerated by the system**, never typed by the user. Must be unique,
  URL-safe, and is the public profile identifier: `garikagada.com/<username>`. Once assigned, it
  is never silently reassigned or recycled.
- Passwords hashed with a memory-hard algorithm (e.g. `PasswordHasher<T>`, Argon2id) — plaintext
  or reversible encryption is PROHIBITED.
- Password recovery MUST NOT depend on any out-of-band PII channel (no "email us a reset link").
- **Auth provider: Keycloak**, self-hosted as its own Podman container via the Aspire AppHost.
  No third-party IdP or auth-as-a-service, in any form.
- Keycloak realm config: username = autogenerated handle; email field left unset, never
  required/verified; no identity brokering / social login (both would pull in third-party PII).
- OIDC claims that imply PII (`email`, `email_verified`, `phone_number`, `name`, `given_name`,
  `family_name`, etc.) MUST NOT be requested, stored, or issued in any token.
- Tokens are scoped to the account's `Uid` (GUID) as the subject — never the internal sequential
  integer ID. Enforce via a custom Keycloak protocol mapper or an equivalent BFF-side claim
  transform.
- The BFF validates Keycloak-issued OIDC tokens at the edge; it does not host its own
  authorization server.

## Self-hosting & infrastructure

- Everything (services, databases, config values) MUST be declared in the **Aspire AppHost**.
  Local dev MUST be a 1:1 functional replica of production — no manual setup steps, no
  environment-only services.
- **Podman only** for all containers. Docker MUST NOT be a required dependency.
- **PostgreSQL** is self-hosted as a Podman container via Aspire — no managed cloud DB, ever.
- Secrets in local dev must mirror the *shape* of production secrets (same key names/structure;
  values may differ).
- **No external SaaS of any kind** — not for auth, not for the database, not for observability.

## Frontend (Angular)

- **Angular Material only** for UI components — no other component library without a
  constitution amendment.
- SCSS is allowed only for: Material theming (`@use '@angular/material' as mat`),
  component-scoped layout overrides Material primitives can't express, and `@media` breakpoint
  overrides. Every such block needs an inline comment justifying it.
- Absolutely PROHIBITED: `::ng-deep` (any form, any file), inline `[style]`/`style` attribute
  bindings, global CSS bypassing the theming API, utility-class CSS frameworks.
- Design language: **peacock-feather-derived M3 palette** (iridescent teal/indigo/emerald/gold),
  feather "eye" motif for letter/envelope iconography. All colors MUST resolve to M3 tokens —
  no hard-coded hex values anywhere, even for the peacock palette.
- All shared state MUST go through **NgRX Signal Store**. No imperative state mutation outside
  store actions. Async flows use RxJS Observables/Angular Signals — no raw Promise chains in
  component code.

## Real-time delivery

- **SignalR is the sole real-time transport** (`@microsoft/signalr` client ↔ BFF-hosted hub).
  New kagada (direct or world-distributed) must reach an open session live — no polling as a
  substitute.
- Hub contracts (hub name, method names, payload shapes) MUST be documented in
  `specs/<feature>/contracts/` before any frontend integration begins.

## Backend architecture

- **BFF-only frontend access**: Angular talks exclusively to `GariKaagada.BFF`. Direct
  frontend → backend API calls are PROHIBITED. The backend API is internal-only.
- BFF responsibilities: SignalR hub host, auth-edge token validation (Keycloak), request
  forwarding. No business logic duplicated between BFF and API.
- BFF → API transport: internal HTTPS or gRPC only. No raw SignalR proxying to the backend.
- **Mandated project layout** (strict layering — no project references a layer above itself):
  - `GariKaagada.BFF` → refs `.BFF.Business`, `.Contracts`
  - `GariKaagada.BFF.Business` → refs `.Contracts`
  - `GariKaagada.Api` → refs `.Api.Business`, `.Contracts`
  - `GariKaagada.Api.Business` → refs `.Api.Data`, `.Contracts`
  - `GariKaagada.Api.Data` → refs `.Contracts` (shared value types only)
  - `GariKaagada.Contracts` → zero dependencies on other solution projects; all
    DTOs/gRPC/SignalR payload types live here, nowhere else
  - `GariKaagada.AppHost` — Aspire AppHost, declares all services/containers
  - `GariKaagada.MigrationWorker` — runs EF Core migrations at startup, before API accepts
    traffic (API itself MUST NOT call `Database.Migrate()`)

## Kagada distribution algorithm

- A "to the world" kagada MUST be delivered to **exactly 5 recipients per calendar day**
  (system-wide, per letter) — never more, never silently short, except when the eligible pool
  is smaller than 5.
- Recipient selection lives behind a single swappable interface (e.g.
  `IKagadaDistributionStrategy`) so uniform-random selection can later be replaced by
  weighted/ML/AI strategies without touching call sites.
- Every distribution decision MUST be logged with structured detail: letter id, candidate pool
  size, selected recipients, strategy name/version.
- Distribution logic lives in `GariKaagada.Api.Business` only — never duplicated in the BFF.

## Recipient consent & control

- Users can disable **direct** kagada (letters addressed to them) at any time; once disabled,
  the **backend** rejects new direct kagada before persistence — not just a client-side filter.
- Direct-kagada opt-out and world-kagada opt-out are **two independent toggles** — never
  conflated into one flag.

## Contracts, validation & codegen

- **FluentValidation in `GariKaagada.Contracts` is the single source of truth** for all
  validation rules. Angular validators are a UX-only replica; if they diverge, the C# rule
  wins and Angular must be updated to match.
- Validators must be exhaustive (not just `NotEmpty()` on every field). DI registration happens
  in `GariKaagada.BFF` and `GariKaagada.Api` (defense-in-depth); validator classes live only in
  `Contracts`.
- **TypeScript types/API clients are auto-generated via NSwag** from the BFF's OpenAPI spec —
  hand-written duplicate TypeScript types are PROHIBITED. Generation runs in the build pipeline;
  generated files are committed.
- For rules OpenAPI can't express, hand-write an Angular validator with a traceability comment:
  `// Mirrors: GariKaagada.Contracts.<ValidatorClass>.<RuleName>`.
- **No AutoMapper/Mapster/TinyMapper** or similar — all object mapping is explicit, hand-written
  (static factory methods / extension methods).

## Observability & logging

- `ILogger<T>` exclusively in application code — no direct static logging calls, no string
  interpolation in log messages (structured templates with named placeholders only).
- Exceptions are never swallowed — every catch logs at `Error` with the full exception object.
- API error responses to the frontend: status code + user-readable message + OpenTelemetry
  trace ID only — never a stack trace, exception type, or internal service name.
- **Observability backend: SigNoz**, self-hosted via Podman/Aspire — the sole mandated OTLP
  backend (no Grafana/Jaeger/other without a constitution amendment).
- W3C `traceparent` propagated on all inter-service calls. Angular has a global `ErrorHandler`
  reporting to a BFF logging endpoint, plus an HTTP interceptor injecting `traceparent`.

## Data & EF Core conventions

- **No raw SQL** (`FromSqlRaw`, `ExecuteSqlRaw`, etc.) — EF Core LINQ only.
- **PostgreSQL covers both relational and document-shaped data**: relational tables +
  Fluent API relationships for structured data; native `jsonb` columns (via EF Core owned/JSON
  column support) for semi-structured data. No separate NoSQL engine.
- Entities: `private set`/`init` property setters only; creation via a `public static` factory
  method; mutations via named instance methods — never direct property assignment.
- Every entity inherits `AuditableEntity`: `int Id`, `Guid Uid` (the only ID ever exposed
  externally), `bool IsActive` (soft-delete — **hard deletes PROHIBITED**),
  `CreatedOn/CreatedById/CreatedBy`, `UpdatedOn/UpdatedById/UpdatedBy` — all auto-populated by a
  `SaveChangesInterceptor`.
- Lazy loading enabled globally (`UseLazyLoadingProxies()`), but PROHIBITED inside loops/
  iteration — use explicit `Include` (eager loading) there, with a comment explaining why.
- A `SaveChangesInterceptor` writes an append-only `AuditLog` row on every INSERT/UPDATE/DELETE.

## Interface-driven design

- Every non-trivial class in `*.Business` and `.Api.Data` projects implements a co-located
  interface (`Services/IFooService.cs` next to `Services/FooService.cs`, not a separate
  `Interfaces/` folder). Only ASP.NET Core controllers are exempt.
- DI registrations and constructor parameters reference the interface, never the concrete type.
  No direct `new ConcreteService(...)` outside DI/factory wiring.

## Code quality gates

- `.NET`: `<TreatWarningsAsErrors>true</TreatWarningsAsErrors>` + `<Nullable>enable</Nullable>`
  on every project. **Angular**: `"strict": true` + `strictTemplates: true`.
- No unjustified suppressions: `#pragma warning disable`, `[SuppressMessage]`, `// @ts-ignore`,
  `// eslint-disable[-next-line]`, or the null-forgiving `!` operator all require an inline
  comment proving the suppression/assumption is safe.
- All `public`/`internal` .NET APIs need XML doc comments; exported Angular
  services/components/store features need JSDoc.
- No commented-out code, no `// TODO` merged to main (convert to a tracked issue first).
- A slow feature with excellent code beats a fast one with shortcuts — quality shortcuts get
  reverted/refactored before the feature counts as done.
- **Official documentation over subjective preference**: verify library/API/config choices
  against official docs before adopting them into code, specs, or plans. Don't rely solely on
  memory, blog posts, or tutorials when official docs are checkable. If docs are silent/
  ambiguous/stale, say so explicitly and resolve it via direct verification or a documented,
  auditable rationale — don't silently assume.
- **No shortcuts**: if one is genuinely unavoidable, justify it in the plan's Complexity
  Tracking table — the same escalation path as any other complexity exception.

## Testing policy

- **No test projects of any kind** — no unit, integration, or e2e tests, for frontend or
  backend. This is a deliberate, explicit policy, not an oversight.
- No test files (`.spec.ts`, `*Tests.csproj`, `tests/`, `__tests__/`) and no test libraries
  (xUnit, NUnit, MSTest, Jest, Jasmine, Karma, Playwright, etc.) may be added as dependencies.
  A PR containing test files gets rejected.
- Quality instead comes from: strong typing (TS strict mode, C# nullable refs), compile-time
  safety (interfaces, FluentValidation, EF Core model validation), linting/formatting, and
  architectural pattern enforcement (layering, factory pattern, no raw SQL).
