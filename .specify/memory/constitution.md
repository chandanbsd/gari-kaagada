# GariKaagada Constitution

## Core Principles

### I. Zero-PII, Password-Only Authentication (NON-NEGOTIABLE)

Account creation MUST require exactly one user-supplied credential: a password. No email
address, phone number, real name, or other personally identifiable information MUST ever be
collected, requested, or stored anywhere in the system — not at signup, not for password
recovery, not for analytics. Every account's username MUST be generated automatically by the
system (never typed by the user), MUST be unique, MUST be URL-safe, and serves as the public
profile identifier at `garikagada.com/<username>`. Once assigned, a username MUST NOT be
silently reassigned or recycled to a different account. Passwords MUST be hashed with a
memory-hard, industry-standard algorithm (e.g., ASP.NET Core Identity's `PasswordHasher<T>`
or Argon2id) — plaintext or reversibly-encrypted password storage is PROHIBITED. Because there
is no email or phone number on file, password recovery MUST NOT depend on any out-of-band PII
channel; any recovery mechanism (e.g., a recovery code shown once at signup) MUST be designed
so it never requires collecting PII after the fact.

Authentication and token issuance MUST be implemented using **Keycloak**, self-hosted as a
dedicated Podman container provisioned through the Aspire AppHost (Principle II — no
third-party identity provider or auth-as-a-service of any kind; self-hosting Keycloak keeps
identity data within the operator's own infrastructure). GariKaagada's Keycloak realm MUST be
configured so account creation requires only a password: the autogenerated username (never an
email address) is the Keycloak username, the email field MUST be left unset and MUST NOT be
required or verified, and no Keycloak identity-brokering or social-login integration (which
would pull in third-party PII) is permitted. The standard OIDC identity claims that imply PII
(`email`, `email_verified`, `phone_number`, `name`, `given_name`, `family_name`, etc.) MUST NOT
be requested, stored, or issued in any token — the BFF MUST request only the minimal
scopes/claims needed to identify the account. Refresh and access tokens MUST be scoped to the
account's `Uid` (Principle XIII) as the sole subject identifier — never a sequential integer
ID; this MUST be enforced via a custom Keycloak protocol mapper (or an equivalent claim
transformation at the BFF) that issues `Uid` as the token's `sub` claim.

**Rationale**: Anonymity is the product's core trust promise — GariKaagada exists so people can
send and receive letters without exposing who they are. Any PII field, however small, becomes a
deanonymization vector and a data-breach liability the product's premise cannot tolerate.

### II. Self-Hosted Sovereignty & Infrastructure as Code (NON-NEGOTIABLE)

GariKaagada MUST run entirely on infrastructure the operator controls. Every service,
database, and configuration value MUST be declared in the .NET Aspire AppHost project; the
local development environment MUST be a 1:1 functional replica of the production topology —
no manual setup steps, no environment-only services, and no "works on my machine" exceptions.
All container images MUST run via **Podman**; Docker MUST NOT be a required dependency.
PostgreSQL MUST be self-hosted as a Podman container wired through Aspire — no managed
cloud database, and no third-party auth-as-a-service provider, is permitted in any
environment (local, CI, or production). Secrets in local development MUST mirror the shape of
production secrets (values may differ; key names and structure MUST NOT).

**Rationale**: A letter-writing service predicated on user anonymity cannot outsource its data
to a third party it does not control — self-hosting is not a deployment preference here, it is
the trust boundary the entire product depends on.

### III. Angular Material + Controlled SCSS with Peacock-Feather Design Language (NON-NEGOTIABLE)

All UI MUST be built on Angular Material components and its theming system; no other UI
component library may be used without a constitution amendment. **SCSS is permitted** only
for: Angular Material theming (customizing the Material Design 3 color scheme, typography,
and tokens via `@use '@angular/material' as mat`), component-scoped layout overrides (margin,
padding, gap, flex/grid) that cannot be expressed with Material primitives, and responsive
`@media` overrides at Material's named breakpoints. Every such SCSS block MUST carry an inline
comment explaining why a Material primitive was insufficient. **Absolutely PROHIBITED — no
exceptions**: `::ng-deep` in any form, inline styles via `[style]` bindings or the `style`
attribute, global CSS overrides that bypass the theming API, and utility-class CSS libraries
(Tailwind, Bootstrap, etc.).

The visual identity MUST draw from **peacock feather** motifs (the "gari" in "Gari Kaagada"):
the M3 color palette MUST be derived from peacock feather hues (iridescent teal, indigo,
emerald, and gold accents), the "eye" motif of a peacock feather MUST inform iconography for
letters (kagada) and envelopes, and decorative use of feather/eye imagery MUST still resolve to
M3 color roles and tokens — hard-coded hex values to achieve the peacock palette are
PROHIBITED; the palette MUST be defined once in the M3 theme and consumed via tokens
everywhere.

**Rationale**: Restricting SCSS to theming and layout-only overrides keeps the UI consistent,
accessible, and dark-mode-capable while still allowing the ergonomic control needed to express
a distinctive, culturally-rooted brand identity within Material's design system rather than
around it.

### IV. Reactive-First Architecture (NON-NEGOTIABLE)

All shared application state MUST be managed through NgRX Signal Store. Imperative state
mutations outside of store actions are PROHIBITED. Async data flows MUST use RxJS Observables
or Angular Signals; raw Promise chains in component code are PROHIBITED. Local component state
(transient UI flags, form control state) is permitted only when the data is not consumed by any
other component or service.

**Rationale**: A single reactive state layer keeps components as pure projections of state and
makes inbox/delivery updates (new kagada arriving, daily world-kagada allotment changing)
predictable to reason about and free of cascading change-detection bugs.

### V. Real-Time Delivery via SignalR

New kagada arriving in a user's inbox — whether a direct letter or a world-distributed one —
MUST be delivered to an open session in real time, without a page reload or manual refresh.
**SignalR is the sole mandated real-time transport**: the Angular frontend MUST use
`@microsoft/signalr` to connect to the BFF's SignalR hub, and the BFF MUST relay delivery
events from the backend API via SignalR. HTTP polling as a substitute for push-based delivery
notifications is PROHIBITED. Hub contracts (hub name, method names, payload shapes) MUST be
documented in `specs/<feature>/contracts/` before any frontend hub integration begins.

**Rationale**: SignalR is the idiomatic real-time transport for ASP.NET Core with a mature
Angular client, and mandating a single transport avoids fragmented choices (raw WebSockets,
SSE, polling) across the inbox, direct-kagada, and world-kagada delivery features.

### VI. BFF Architecture (NON-NEGOTIABLE)

The Angular frontend MUST connect exclusively to the BFF (Backend For Frontend) ASP.NET Core
project. Direct calls from the frontend to the backend API are PROHIBITED. The BFF is the sole
internet-facing .NET service; the backend API MUST be network-isolated to internal service
communication only. The BFF is responsible for: hosting the SignalR hub the frontend connects
to, enforcing authentication at the edge by validating OIDC tokens issued by the self-hosted
Keycloak authorization server (Principle I), and forwarding domain requests to the backend API.
All BFF → backend API calls MUST use internal HTTPS or gRPC; raw SignalR proxying from BFF to
backend is PROHIBITED. No business logic MUST be duplicated between BFF and backend — the BFF
acts as a thin translation and security layer.

**Rationale**: The BFF pattern keeps the public attack surface minimal and lets frontend-facing
concerns (auth edge, SignalR fan-out) evolve independently of the core domain API, which is
important for a service whose entire value proposition is protecting user anonymity.

### VII. Layered .NET Project Architecture (NON-NEGOTIABLE)

All .NET code MUST follow a strict layered project structure. No project may reference a
project in a layer above its own. The mandated solution layout is:

*BFF tier*
- `GariKaagada.BFF` — ASP.NET Core Web API; entry point, controllers, SignalR hub; references
  `GariKaagada.BFF.Business` and `GariKaagada.Contracts` only.
- `GariKaagada.BFF.Business` — class library; BFF business logic, auth forwarding, service
  orchestration; references `GariKaagada.Contracts` only.

*Backend API tier*
- `GariKaagada.Api` — ASP.NET Core Web API; entry point and controllers; references
  `GariKaagada.Api.Business` and `GariKaagada.Contracts` only.
- `GariKaagada.Api.Business` — class library; all domain logic (accounts, kagada, distribution
  algorithm); references `GariKaagada.Api.Data` and `GariKaagada.Contracts`.
- `GariKaagada.Api.Data` — class library; EF Core `DbContext`, entity classes, migrations,
  repository implementations; references `GariKaagada.Contracts` for shared value types only.

*Shared*
- `GariKaagada.Contracts` — class library; all request/response DTOs, gRPC message types, and
  SignalR event payload types crossing the BFF ↔ API boundary. Zero dependencies on other
  solution projects. Neither BFF nor API may define ad-hoc inter-service types outside this
  library.

*Aspire orchestration*
- `GariKaagada.AppHost` — .NET Aspire AppHost; declares all services, containers, and
  references.
- `GariKaagada.MigrationWorker` — Aspire worker service; runs EF Core Migrations at startup
  before the API becomes available.

**Rationale**: Enforcing project-level layer boundaries via `.csproj` references makes illegal
dependencies a compile error, not a code-review finding. The Contracts library eliminates
implicit coupling between services.

### VIII. Kagada Distribution Algorithm Integrity (NON-NEGOTIABLE)

A kagada written "to the world" (not addressed to a specific user) MUST be delivered to exactly
5 recipients per calendar day, system-wide, per authored letter as specified — no user MUST
receive more than the daily cap allows and no letter MUST be silently dropped or delivered
short of its allotment except when the pool of eligible recipients is smaller than the cap. The
recipient-selection strategy MUST be implemented behind a single, swappable interface (e.g.,
`IKagadaDistributionStrategy`) so that the initial uniform-random selection can be replaced by
future weighted, ML-, or AI-driven selection strategies without changing any calling code. Every
distribution decision MUST be logged with enough structured detail (letter id, candidate pool
size, selected recipients, strategy name/version) to reconstruct why a given recipient was
chosen, since this algorithm is the product's core differentiator and must remain auditable and
independently improvable over time. Distribution logic MUST live in
`GariKaagada.Api.Business` and MUST NOT be duplicated or reimplemented in the BFF.

**Rationale**: The "5 random people a day" mechanic is explicitly called out as what
differentiates GariKaagada from other letter/messaging products, and the intent is to keep
iterating on the selection algorithm (including future AI/ML approaches). Locking it behind one
interface and mandating structured audit logging from day one means the algorithm can evolve
without a rewrite and without losing the ability to explain past distributions.

### IX. Recipient Consent & Control (NON-NEGOTIABLE)

Every user MUST be able to disable direct kagada (letters addressed to them by name/URL) at any
time; once disabled, the backend MUST reject new direct kagada to that user before they are
persisted or queued, not merely hide them client-side. A user's direct-kagada preference MUST
NOT affect their eligibility to receive world-distributed kagada (Principle VIII) unless the
user separately opts out of world distribution — these are two independent consent toggles and
MUST be modeled and enforced as such, never conflated into a single flag.

**Rationale**: Letting anyone write to a user's public profile page is core to the product, but
so is giving that user an unambiguous, immediately effective way to stop it — consent controls
that only filter on the client are not real controls.

### X. Code Quality & Maintainability First (NON-NEGOTIABLE — OVERARCHING PRINCIPLE)

This is the highest-priority principle in this constitution. GariKaagada is a
**software-engineering-excellence project first**; a shipped product second. When any decision
creates tension between code quality and delivery speed, **code quality MUST win, without
exception**.

**Zero-warning builds**
- All .NET projects MUST set `<TreatWarningsAsErrors>true</TreatWarningsAsErrors>` and
  `<Nullable>enable</Nullable>`. No warnings are acceptable in any build configuration.
- The Angular project MUST compile with `"strict": true` in `tsconfig.json` and
  `strictTemplates: true` in `angularCompilerOptions`.

**No unjustified suppressions**
- `#pragma warning disable`, `[SuppressMessage]`, `// @ts-ignore`, and
  `// eslint-disable[-next-line]` directives MUST NOT be committed without an inline comment at
  the same location explaining precisely why the suppression is safe.
- The null-forgiving operator (`!`) MUST NOT be used without an inline comment proving null is
  impossible at that point.

**Mandatory documentation**
- All `public` and `internal` .NET APIs MUST have XML documentation comments.
- All exported Angular services, components, and store features MUST have JSDoc comments.

**No committed noise**
- Commented-out code MUST NOT appear in any committed changeset.
- `// TODO` comments MUST NOT be merged to main — they become tracked issues before merge.

**Quality gates over velocity**
- A feature delivered slowly with excellent code is ACCEPTABLE and PREFERRED.
- A feature delivered quickly with quality shortcuts MUST be reverted or refactored before it
  may be considered complete.

**Best practices over shortcuts, official documentation over subjective preference**
- Every non-trivial technical decision (library choice, API usage, configuration, architecture
  pattern) MUST be verified against the official documentation of the technology in question
  before it is adopted or written into code, specs, or plans. Blog posts, tutorials, forum
  answers, and unverified prior knowledge MAY inform a starting hypothesis but MUST NOT be the
  sole basis for a decision when official documentation exists and is checkable.
- Shortcuts, "good enough for now" implementations, or workarounds that knowingly skip a
  documented best practice are PROHIBITED. If a shortcut is genuinely unavoidable, it MUST be
  justified in the plan's Complexity Tracking table, the same escalation path required for
  other complexity exceptions.
- Where official documentation is ambiguous, contradictory, out of date, or silent on a point
  needed for the decision at hand, that gap MUST be stated explicitly (not silently assumed
  away) and resolved either by direct verification against the running system/tool, or by a
  documented decision with rationale that a reviewer can audit later.

**Rationale**: Technical debt incurred at a project's foundation compounds faster than anywhere
else. Establishing and mechanically enforcing zero-tolerance quality standards from commit one
is orders of magnitude cheaper than retrofitting them later. This is a deliberate, informed
choice, not subject to trade-off negotiation.

### XI. Contract-Driven Validation & TypeScript Generation (NON-NEGOTIABLE)

**C# FluentValidation is the single source of truth for all validation rules.** Every
validation rule a user must satisfy before submitting a form EXISTS FIRST in a C#
FluentValidation `AbstractValidator<T>` in `GariKaagada.Contracts`. Angular reactive form
validators are a replica of those rules for UX purposes only; if the two ever diverge, the C#
rule is correct and the Angular rule MUST be updated to match.

Every request payload type in `GariKaagada.Contracts` originating from the frontend MUST have
a co-located `AbstractValidator<T>` in the same project. Validators MUST be exhaustive — thin
validators that only check `NotEmpty()` on every field are PROHIBITED. Validator DI
registration occurs in `GariKaagada.BFF` (frontend-facing) and `GariKaagada.Api`
(defense-in-depth); validator class definitions live exclusively in `GariKaagada.Contracts`.

All TypeScript types and API client code MUST be auto-generated from the BFF's OpenAPI
specification using **NSwag**; hand-written TypeScript duplicating C# contract types is
PROHIBITED. Generation MUST run as part of the build pipeline; generated files MUST be
committed so type regressions appear as diffs in PRs. For rules OpenAPI cannot express
(cross-field constraints, conditional nullability), Angular MUST implement a custom validator
that replicates the FluentValidation rule exactly, carrying a traceability comment:
`// Mirrors: GariKaagada.Contracts.<ValidatorClass>.<RuleName>`.

**AutoMapper and all similar convention-based object-mapping libraries (Mapster, TinyMapper,
etc.) are PROHIBITED.** All object-to-object mapping MUST be explicit, hand-written code
(static factory methods or extension methods).

**Rationale**: Forms are a trust boundary — deriving Angular validators from the same
FluentValidation rules the backend enforces eliminates frontend/backend disagreement, and
explicit mapping fails at compile time instead of silently at runtime.

### XII. Observability & Structured Logging (NON-NEGOTIABLE)

All application code MUST use `ILogger<T>` exclusively; direct static logging calls are
PROHIBITED. All log messages MUST use structured message templates with named placeholders —
string interpolation in log calls is PROHIBITED. Exceptions MUST NOT be swallowed: every caught
exception MUST be logged at `Error` level with the full exception object. API error responses
sent to the frontend MUST NOT include stack traces, exception type names, or internal service
names — they MUST return an appropriate HTTP status code, a user-readable message, and the
OpenTelemetry trace ID of the request.

All .NET services MUST export traces, metrics, and logs via OTLP to **SigNoz** — the project's
sole mandated observability backend — self-hosted and provisioned as one or more Podman
containers wired through the Aspire AppHost. No external/managed observability SaaS, and no
alternative OTLP backend (Grafana Tempo/Loki, Jaeger, or any other), is permitted in any
environment without a constitution amendment (Principle II). W3C TraceContext (`traceparent`)
MUST be propagated on all
inter-service calls. The Angular app MUST implement a global `ErrorHandler` that reports
unhandled errors to a BFF logging endpoint, and an HTTP interceptor MUST inject `traceparent`
on all outgoing requests.

**Rationale**: Abundant, structured logging is the primary debugging tool for a real-time,
async-heavy application; the trace ID is the safe bridge between a user-reported error and the
full diagnostic context, and self-hosting observability keeps it consistent with the project's
sovereignty requirement.

### XIII. Entity Framework Core Conventions (NON-NEGOTIABLE)

**No raw SQL.** `FromSqlRaw`, `FromSqlInterpolated`, `ExecuteSqlRaw`, `ExecuteSqlInterpolated`,
and `Database.ExecuteSql*` are PROHIBITED. All data access MUST go through EF Core LINQ.

**Relational and document data both live in PostgreSQL.** Structured, relational data (users,
kagada metadata, distribution records) MUST use standard relational tables and Fluent API
relationships in `OnModelCreating`. Semi-structured or schema-flexible data (e.g., letter body
content, evolving metadata) MUST use PostgreSQL's native `jsonb` columns mapped via EF Core's
owned/JSON column support — a separate NoSQL database engine MUST NOT be introduced; PostgreSQL
is the single source of truth for both relational and document-shaped data.

**Entity encapsulation.** All entity property setters MUST be `private set` or `init`. Entity
creation MUST use a `public static` factory method; post-creation mutations MUST go through
named instance methods, never direct property assignment.

**Universal `AuditableEntity` base class.** Every entity MUST inherit from `AuditableEntity`
and expose: `int Id`, `Guid Uid` (the only ID ever exposed externally, preventing enumeration of
sequential integer IDs), `bool IsActive` (soft-delete; hard deletes are PROHIBITED),
`CreatedOn`/`CreatedById`/`CreatedBy`, and `UpdatedOn`/`UpdatedById`/`UpdatedBy`, all populated
automatically by a `SaveChangesInterceptor`.

**Loading strategy.** Lazy loading MUST be enabled globally via `UseLazyLoadingProxies()`.
Using lazy loading inside a loop or iteration is PROHIBITED (N+1 prevention) — such call sites
MUST use eager loading with explicit `Include`, documented with an inline comment.

**Audit history.** A `SaveChangesInterceptor` MUST write an append-only `AuditLog` row on every
INSERT, UPDATE, and DELETE, storing entity type/id/uid, action, changed properties, and
old/new values as JSONB.

**Rationale**: The dual-key pattern (`int Id` + `Guid Uid`) keeps joins fast while exposing
non-guessable identifiers — critical when the product's entire premise is protecting user
identity. Using PostgreSQL `jsonb` for document-shaped data avoids operating a second database
engine, which would violate the self-hosting simplicity this project depends on.

### XIV. Interface-Driven Design (NON-NEGOTIABLE)

Every non-trivial class in `GariKaagada.BFF.Business`, `GariKaagada.Api.Business`, and
`GariKaagada.Api.Data` MUST implement a corresponding interface, co-located in the same file
location and namespace as its implementation (e.g., `Services/IKagadaService.cs` alongside
`Services/KagadaService.cs`) — a separate `Interfaces/` folder is PROHIBITED. The **sole
exception** is ASP.NET Core controllers. All DI registrations and constructor parameters MUST
reference the interface, never the concrete type; direct instantiation of service classes
outside factory/DI wiring is PROHIBITED.

**Rationale**: Interface-driven design makes every dependency a seam, and enforces Dependency
Inversion at the architectural level — higher layers depend on abstractions, never on
concretions in lower layers.

### XV. No Test Projects (NON-NEGOTIABLE)

**No unit tests, integration tests, e2e tests, or test files of any kind MUST be written or
committed to this repository — for either the frontend or the backend.** This is an explicit,
unconditional architectural decision. Quality is achieved instead through: self-documenting
code with intention-revealing names (Principle X), strong type systems (TypeScript strict mode,
C# nullable reference types), compile-time safety (interface contracts, FluentValidation, EF
Core model validation), automated linting/formatting, and architectural pattern enforcement
(layer boundaries, entity factory pattern, no raw SQL). No test project files (`.spec.ts`,
`*Tests.csproj`, `tests/`, `__tests__/`) MUST be created, and no test libraries MUST be added as
dependencies. Pull requests containing test files MUST be rejected.

**Rationale**: A deliberate, informed decision that the return on investment for AI-generated
unit tests in this project does not justify the maintenance overhead they create. Code quality
is the responsibility of design, the type system, linting, and code review — not a test suite
mirroring the implementation.

## Technical Constraints

- **Frontend**: Angular (latest stable LTS), Angular Material, RxJS, NgRX Signal Store, Angular
  Signals, `@microsoft/signalr`, TypeScript, SCSS. No other UI or state libraries without a
  constitution amendment.
- **Real-time transport**: SignalR. Frontend uses `@microsoft/signalr`; BFF and backend use
  ASP.NET Core SignalR hubs.
- **Backend services** (both latest stable):
  - *BFF*: ASP.NET Core Web API, C# — internet-facing, hosts SignalR hub, validates auth at the
    edge, calls backend API via internal HTTPS or gRPC.
  - *Backend API*: ASP.NET Core Web API, C#, Entity Framework Core — internal only, owns
    domain logic, the kagada distribution algorithm, and database access.
- **BFF → API transport**: Internal HTTPS or gRPC. Raw SignalR proxying to the backend is
  PROHIBITED.
- **Orchestration**: .NET Aspire (latest stable). All services, containers, environment
  variables, and inter-service references MUST be declared in the Aspire AppHost project.
- **Containers**: Podman. Docker MUST NOT be required.
- **Database**: Self-hosted PostgreSQL, provisioned as a Podman container via Aspire, used for
  both relational tables and `jsonb` document-style data (Principle XIII). No managed cloud DB
  in any environment.
- **Auth**: **Keycloak**, self-hosted as a dedicated Podman container provisioned via the
  Aspire AppHost, configured with a custom realm for the password-only credential scheme
  (Principle I). The BFF validates Keycloak-issued OIDC tokens at the edge (Principle VI). No
  third-party identity provider or auth-as-a-service dependency of any kind — self-hosted
  Keycloak is the sole identity provider.
- **SCSS policy**: Permitted only for Angular Material theming, responsive layout overrides,
  and component-scoped layout properties (Principle III). `::ng-deep`, inline styles, and
  utility-class CSS libraries are PROHIBITED. Hard-coded colors outside the peacock-feather M3
  theme tokens are PROHIBITED.
- **Design system**: Google Material Design 3, themed with a peacock-feather-derived palette
  (Principle III). Angular Material is the implementation vehicle.
- **No testing libraries**: No test framework (xUnit, NUnit, MSTest, Jest, Jasmine, Karma,
  Playwright, etc.) MUST be installed, referenced, or configured (Principle XV).
- **Validation**: FluentValidation is pre-approved for `GariKaagada.Contracts` (validator
  definitions) and `GariKaagada.BFF` / `GariKaagada.Api` (DI registration only).
- **TypeScript generation**: NSwag CLI, run as part of the build pipeline; generated files
  committed to the repository.
- **Prohibited libraries**: AutoMapper, Mapster, TinyMapper, and all convention-based
  object-mapping libraries.
- **Observability backend**: **SigNoz**, self-hosted, provisioned as Podman container(s) via
  Aspire (see Principle XII for topology). No external observability SaaS, and no alternative
  OTLP backend, without a constitution amendment.
- **Logging**: `ILogger<T>` exclusively in application code; structured message templates only.
- **Telemetry SDK**: OpenTelemetry .NET SDK; OTLP export; W3C TraceContext propagation on all
  inter-service calls.
- **Schema migrations**: All schema changes MUST be applied via EF Core Migrations, executed by
  `GariKaagada.MigrationWorker` at startup before the API accepts traffic. The API project MUST
  NOT call `Database.Migrate()` itself at runtime.

## Development Workflow

All feature work MUST follow the speckit flow: `/speckit-specify` → `/speckit-plan` →
`/speckit-tasks` → `/speckit-implement`. The Constitution Check gate in `plan.md` MUST be
completed and passing before Phase 0 research begins. API contracts and SignalR hub contracts
MUST be defined in `specs/<feature>/contracts/` before any frontend integration starts. Every
plan MUST include a task for declaring new services or containers in the Aspire AppHost.
Features are delivered incrementally by user story, with each story independently
demonstrable against the full local Aspire stack.

Any change to the kagada distribution algorithm (Principle VIII) MUST document, in the
feature's `plan.md`, the candidate-selection strategy being introduced or modified and how it
preserves the 5-per-day cap and audit logging requirements.

## Governance

This constitution supersedes all other practices, style guides, and conventions. Any conflict
between this document and another guideline resolves in favor of this constitution.

**Principle X (Code Quality & Maintainability First) supersedes all other principles when they
conflict.** If following any other principle would require a quality shortcut, the shortcut is
not permitted — the other principle must be satisfied in a quality-compliant way or the feature
must be deferred until it can be.

Amendments require:
1. A documented rationale explaining why the existing principle is insufficient.
2. A semantic version bump (MAJOR for principle removal/redefinition, MINOR for additions,
   PATCH for clarifications).
3. Propagation of the change to all affected templates and an updated Sync Impact Report.
4. Review and explicit acceptance before the next feature spec is opened.

All PRs MUST verify compliance with Principles I (Zero-PII Auth), II (Self-Hosted Sovereignty),
III (Angular Material + Peacock-Feather Design), IV (Reactive-First), V (Real-Time via
SignalR), VI (BFF Architecture), VII (Layered .NET Architecture), VIII (Kagada Distribution
Algorithm Integrity), IX (Recipient Consent & Control), X (Code Quality & Maintainability
First), XI (Contract-Driven Validation & TypeScript Generation), XII (Observability &
Structured Logging), XIII (EF Core Conventions), XIV (Interface-Driven Design), and XV (No Test
Projects) in the Constitution Check section of `plan.md`.

Principle X applies to every line of every PR. Any new Contract DTO touching the frontend
boundary MUST have a co-located FluentValidation validator (Principle XI). Any new service
endpoint MUST include structured logging for the happy path and all error branches (Principle
